# AIOps 智能运维 Agent 项目设计文档

## 一、项目概述

### 1.1 背景与目标
本项目源于横向项目"光网络故障定位技术研究"。快速精准的故障定位技术，是保障大量网络服务正常运行和保障业务数据完整可靠传输的基础。

**痛点问题：**
- 光网络单断点连锁海量告警
- 故障传导复杂
- 人工逐层排查耗时长
- 多维度指标难定位断点
- 链路故障直接造成上层业务、仓储系统传输中断

**解决方案：** 打造智能 Agent，让其能自动回答常见问题，并在故障发生时主动进行初步排查。

### 1.2 核心能力
1. **RAG 知识库**：基于 Eino 框架设计搭建，作为团队文档管理，将日常积累的文档（告警处理手册、技术方案、错误码文档）转化为可被 AI 高效检索的向量
2. **对话 Agent**：基于大模型 + 知识库的智能交互系统，挡掉高频重复咨询，加速问题解决
3. **运维 Agent**：依托自研 DC 因果根因定位算法做故障溯源，联动日志、监控、知识库系统，自动提取关键参数，生成结构化故障排查报告

## 三、故障定位技术详解

### 3.1 光网络故障场景特殊性分析

光网络环境下的故障具有鲜明的链式传导特征，这是本项目需要解决的核心挑战：

| 特性 | 描述 | 技术影响 |
|------|------|----------|
| 单断点海量连锁告警 | 一处光纤断点触发上百个关联设备告警 | 告警风暴，人工无法筛选 |
| 故障传导隐蔽 | 断点不直接出现在故障设备上，通过光链路层层传导 | 根因与表象物理分离 |
| 跨域依赖复杂 | 光层→传输层→IP层→应用层多层级传导 | 难以跨维度追溯 |
| 指标关联度低 | 大量正常指标伴随少量异常指标混杂 | 多维度定位困难 |
| 业务影响直接 | 链路瞬间中断导致WMS仓储系统、上层业务全部断连 | 要求秒级定位 |

**典型告警传播链路示例：**
```
光纤物理断点 → 光功率告警 → 光模块告警 → 端口Down告警 → 
OSPF邻居断连 → BGP路由撤销 → 服务丢包告警 → 应用超时告警
```
一个真实的物理断点，最终向上层扩散出几十到上百条不同类型的告警，人工从最上层的应用告警反向回溯到物理断点平均耗时超过30分钟。

### 3.2 自研 DC 因果根因定位算法

DC（Dependency Chain，依赖链）算法是项目的核心创新，专门针对光网络链式故障传导场景设计：

#### 算法核心思想
基于**因果依赖图 + 时序关联度 + 传播距离权重**三位一体的根因评分模型，自动从海量告警中定位最初的单断点。

#### 算法输入
DC 算法采用多源异构数据融合作为输入，包含四大维度：
- `G`：网络拓扑依赖图（顶点为光设备/交换机/服务，边为物理连接/依赖关系）
- `A`：全量告警集合（字段：告警ID、时间戳、告警类型、关联对象、告警级别）
- `M`：时序监控指标集合（CPU、内存、光功率、端口流量、丢包率等，按时间序列采样）
- `L`：系统与应用日志集合（关键字段：日志时间、日志级别、服务名、日志内容）
- `T`：故障分析时间窗口（可配置，默认故障前后 15 分钟）

#### 三步核心计算过程

**第一步：构建告警因果子图**
```
G_A = 所有包含告警顶点的连通子图
```
过滤掉无告警的正常节点，大幅缩小分析范围。

**第二步：时序传播距离加权**
对每条告警计算相对于最早告警的传播时延：
```
weight(a) = 1 / (Δt(a) + ε)
Δt(a) = t(a) - t_first_alarm
```
距离最早告警时间越近的告警，权重越高，指向根因的可能性越大。

**第三步：根因置信度综合评分**
```
Score(v) = α * Centrality(v) + β * EarlyTime(v) + γ * TopologyDist(v)
```
其中：
- `Centrality(v)`：节点在告警子图中的图中心性得分，度越高得分越高
- `EarlyTime(v)`：告警发生的时间越早得分越高
- `TopologyDist(v)`：到所有其他告警节点的平均最短路径距离越小得分越高
- `α, β, γ` 为可训练的超参数（预设值 0.4, 0.4, 0.2）

取 `Score(v)` 最高的前 3 个节点作为候选根因输出。

### 3.3 告警降噪与关联分析

在执行因果根因定位之前，系统首先对原始告警流进行预处理：

#### 1. 告警去重与合并
- 5分钟内相同源设备、相同告警类型的重复告警合并为一条
- 保留首次触发时间和最新触发时间，避免相同告警刷屏

#### 2. 告警抑制
- 当父节点发生故障时，自动抑制其所有子节点派生告警
- 例如：光链路断连后其子设备的所有衍生告警全部标记为"派生告警"，不参与根因计算

#### 3. 告警分组
基于时间窗口和拓扑邻近性自动分组：
- 时间距离 < 2分钟，拓扑距离 < 3跳的告警自动聚合为同一个故障事件
- 实现"一次故障，一个事件"，避免将连锁告警当成多个独立故障

### 3.4 全链路故障传导追踪

定位到根因节点后，系统自动反向生成完整故障传导路径：

```
根因节点 → 一级传导告警 → 二级传导告警 → 最终业务影响
```

输出完整的传导链信息包含：
- 故障源精准定位（物理位置、设备ID、端口号）
- 传导路径上的每一个关键节点和时间戳
- 受影响的服务清单、影响范围估算
- 传导延迟分析（从根因发生到业务感知耗时）

**传导追踪示例输出：**
```
# 故障传导链路报告

根因：东莞机房 1号ODF架 第12芯光纤中断 (2025-06-08 14:23:12)
  ↓
一级传导：SW02 光模块1 光功率过低告警 (14:23:13, 延迟 1s)
  ↓
二级传导：SW02 GE0/12 端口物理Down (14:23:14, 延迟 2s)
  ↓
三级传导：OSPF 邻居 10.0.0.5 断连 (14:23:16, 延迟 4s)
  ↓
四级传导：BGP 路由 192.168.100.0/24 撤销 (14:23:18, 延迟 6s)
  ↓
五级传导：应用服务器 192.168.100.10 连接超时 (14:23:35, 延迟 23s)
  ↓
最终影响：WMS仓储系统上传服务中断
```

### 3.5 故障定位完整执行流程

结合 Agent 的 Plan-Execute-Replan 工作流，完整的故障定位全流程：

| 阶段 | 执行动作 | 工具调用 | 输出产物 |
|------|----------|----------|----------|
| 1. 告警采集 | 拉取最近告警，去重降噪 | MCP 告警查询工具 | 清洗后的告警集合 |
| 2. 拓扑加载 | 加载相关区域网络拓扑 | MCP 拓扑查询工具 | 告警因果子图 |
| 3. DC算法计算 | 运行因果根因定位 | 内置核心引擎 | Top3 候选根因节点 |
| 4. 证据补充 | 针对候选根因定向查询监控指标、系统日志 | MCP 监控+日志工具 | 根因验证证据链 |
| 5. RAG经验匹配 | 检索历史相似故障处理案例 | RAG知识库检索工具 | 历史处理参考方案 |
| 6. 生成诊断报告 | 融合所有证据生成结构化报告 | LLM 汇总输出 | 完整故障排查报告 |

### 3.6 光网络单断点定位实战示例

**输入告警集合（共 47 条连锁告警）：**
```
告警1: 东莞机房 ODF-01-12 光纤信号丢失  14:23:12  严重
告警2: SW02 光模块1 Rx Power < -28dBm  14:23:13  严重
告警3: SW02 GE0/12 Link Down            14:23:14  重要
...
告警47: WMS 上传服务超时率 > 90%        14:23:42  严重
```

**DC 算法输出根因得分排序：**
```
节点1: ODF-01-12 (光纤断点)  Score = 0.97  →  推荐根因
节点2: SW02 (交换机)         Score = 0.62
节点3: WMS-Server (应用)    Score = 0.31
```

**最终 Agent 输出报告结论：**
> 根因确认：1号ODF架第12芯光纤物理断裂。
> 
> 证据链：
> 1. 该告警是全量告警集合中最早发生的第一条告警
> 2. 所有其他告警在拓扑上均依赖于此链路
> 3. 该光纤端口最近30分钟光功率指标从-6dBm直接掉到-40dBm以下
> 4. 历史上同位置发生过完全相同的光缆被挖断事件
> 
> 处理建议：通知外线工程队立即赶赴现场熔接光缆，预计修复时间 30 分钟。

## 四、核心模块详解

### 4.1 Planner 模块

**核心职责：** 接收用户输入，从 RAG 知识库查经验，获取所有可用工具，生成结构化排查计划

**执行流程：**

1. **接收用户输入**
   ```
   用户输入："系统 CPU 占用 100%，帮忙排查一下"
   Planner 拿到这个 input
   ```

2. **从 RAG 知识库查经验（最关键）**
   ```python
   context_str = await retrieve_knowledge.ainvoke({"query": input_text})
   ```
   - 自动查历史故障手册、CPU / 磁盘排查 SOP、告警处理文档
   - 让计划不是瞎编，而是基于团队真实经验

3. **获取所有可用工具**
   - 本地工具：时间、知识库检索
   - MCP 工具：查 CPU、查磁盘、查内存、查日志、查监控、查接口...

4. **LLM 生成结构化排查计划**
   ```python
   planner_chain = planner_prompt | llm.with_structured_output(Plan)
   ```

   **输出格式：**
   ```json
   {
     "steps": [
       "步骤1：查询CPU使用率",
       "步骤2：查询进程占用Top10",
       "步骤3：查询系统日志",
       "步骤4：查询磁盘IO是否瓶颈",
       "步骤5：生成排查报告"
     ]
   }
   ```

**示例：**
用户说 "服务异常，CPU 高，磁盘可能满了"，Planner 自动生成：
1. 查询最近 15 分钟 CPU 使用率
2. 查询磁盘使用率与 inode
3. 查询系统错误日志
4. 查询进程占用资源排行
5. 结合知识库经验判断根因
6. 生成排查报告

---

### 4.2 Executor 模块

**核心职责：** 按计划步骤调用工具执行

**执行流程：**

1. **取任务**：从 state 里拿到计划，取出第一个步骤执行
   - 例如：查询当前服务器CPU使用率

2. **加载所有工具**
   - 本地工具：获取时间、检索 RAG 知识库
   - MCP 工具：查 CPU、查磁盘、查内存、查日志、查监控、查接口

3. **LLM 绑定工具 → 判断用什么工具**
   - "查询当前服务器CPU使用率" → 自动决定调用 `get_cpu_metrics` 工具

4. **自动调用工具（ToolNode）**
   - LangGraph 自带 ToolNode 自动执行工具调用，不用手写工具调用逻辑

5. **返回结果**
   - 步骤执行结果存入 `past_steps`
   - 计划删除已完成的第一步
   - 交给下一步继续执行

---

### 4.3 Replanner 模块

**核心职责：** 评估当前状态，决定下一步行动

**三种动作输出：**

1. **respond —— 信息足够，直接生成最终故障报告**
   - 触发条件：
     - 已获取 CPU、磁盘、日志、监控数据
     - 问题根因已经明确
     - 步骤执行 ≥ 3~5 步
   - 行为：停止执行 → 生成结构化排查报告

2. **continue —— 计划没问题，继续执行下一步**
   - 触发条件：计划合理、信息还不够
   - 行为：继续让 Executor 跑

3. **replan —— 原计划错了，重新修改步骤**
   - 触发条件：原计划漏掉关键步骤、工具调用失败
   - 行为：重新生成更合理的计划

**完整执行流程（清晰5步）：**

1. 拿到当前状态：用户问题 + 已执行步骤 + 剩余计划
2. 判断是否步骤太多（保护机制）：超过8步直接强制结束，防止 Agent 无限循环
3. 获取所有可用工具
4. LLM 做决策：respond / continue / replan
5. 执行动作：继续 / 重规划 / 结束

---

### 4.4 MCP 工具调用机制

#### 重试拦截器（最核心）
```python
async def retry_interceptor(...):
```
- 调用查 CPU、查磁盘、查日志工具时如果失败，自动重试3次
- 采用指数退避（等待时间越来越长）
- 最终失败也不会抛错，返回错误提示
- 保证 Agent 在生产环境稳定不死

#### 全局单例客户端
```python
async def get_mcp_client_with_retry():
```
- 整个系统只创建一个 MCP 连接
- 避免重复连接、重复初始化
- 所有模块共用一套连接，高效稳定

#### 自动连接所有 MCP 服务器
```python
DEFAULT_MCP_SERVERS = config.mcp_servers
```
- 一次性连接日志服务器、监控服务器、主机资源服务器
- 所有工具自动加载，Executor 可直接调用

**实际执行场景：**
Executor 执行：
- 查询 CPU 使用率
- 查询磁盘空间
- 查询系统日志
- 查询端口占用

全部通过 MCP 客户端调用真实 API 获取数据。如果调用失败 → 自动重试 3 次 → 还失败 → 返回错误信息 → 不会让整个 Agent 崩溃

## 五、API 接口设计

### 5.1 AIOps 智能诊断接口 `/api/aiops`

**流程：**
1. 前端/用户调用 `/api/aiops`
2. 后端创建会话 session_id
3. 启动智能诊断工作流
4. 实时流式推送每一步状态
5. 最后返回完整故障报告
6. 自动关闭连接

**用户体验：**
用户一句话："服务器 CPU 高、磁盘满了" → 调用接口 → 前端实时看到：
1. 正在生成计划...
2. 正在查询 CPU...
3. 正在查询磁盘...
4. 正在查询日志...
5. 生成最终报告...

**SSE 事件类型：**
1. `status` - 状态更新
2. `plan` - 诊断计划制定完成
3. `step_complete` - 步骤执行完成
4. `report` - 最终诊断报告
5. `complete` - 诊断完成
6. `error` - 错误信息

### 5.2 实时监控告警推送
后端通过监控平台实时检测到新告警后，主动通过 SSE / WebSocket 向前端推送告警消息：
- 后端定时/实时监听监控系统（Prometheus/Zabbix/内部告警平台）的新告警
- 一旦产生新告警，立即通过 SSE 或 WebSocket 推送给前端
- 前端自动弹出告警提示（主机IP、告警类型、告警级别、发生时间）
- 用户点击【一键诊断】，直接调用 `/aiops` 接口开始自动故障定位
- 全程不需要手动输入任何内容

### 5.3 对话接口 `/api/chat`
- 用户提问："CPU 高怎么处理？" "磁盘满了该查哪些指标？"
- 接口自动调用 RAG Agent
- 检索内部手册/历史案例
- 流式返回标准答案

### 5.4 文件上传接口
功能：
1. 用户上传文档（只允许 md/txt，≤10MB）
2. 自动校验格式和文件大小
3. 安全处理文件名（防止路径穿越）
4. 保存文件
5. 自动调用向量库生成索引

特性：
- 文件安全处理：过滤特殊字符，防止恶意文件名
- 自动覆盖更新：文件已存在会自动覆盖，重新生成向量
- 上传与向量索引解耦：即使索引失败，文件仍上传成功
- 支持批量索引目录：一次性把所有文档入库，知识库初始化方便

### 5.5 健康检查接口 `/health`
功能：
- 检查服务基本信息（名称、版本）
- 检查 Milvus 连接是否正常
- 只要 Milvus 挂了 → 整个服务标记为不健康
- 返回状态：正常 200 healthy / 异常 503 unhealthy

## 六、RAG 知识库系统

### 6.1 LLM 工厂类
核心特性：
- 统一创建大模型实例，屏蔽底层厂商差异
- 业务代码与模型厂商解耦
- 支持多种厂商：阿里云 DashScope、OpenAI、Azure OpenAI、本地 Ollama/vLLM
- 统一管理配置：模型名、API Key、base_url
- 支持流式输出、温度设置

### 6.2 Milvus 向量数据库统一客户端
职责：
- 连接 Milvus 向量库
- 自动检查集合是否存在，不存在则自动创建
- 自动检查向量维度是否匹配
- 自动创建向量索引
- 自动加载 collection 到内存
- 提供健康检查
- 上下文管理器安全关闭连接
- 全局单例模式，避免重复连接

### 6.3 文档分割策略
三阶段处理：
1. **第一阶段：按 Markdown 标题分割（# / ##）**
   - 保持章节完整性
   - 语义不被打断
   - 适合运维手册、技术文档
2. **第二阶段：递归字符分割**
   - 太长的章节再切一次
   - 控制片段长度
   - 防止超长
3. **第三阶段：合并过小碎片**
   - 太小的片段没用
   - 自动合并相邻小片
   - 让片段更均匀、更稳定

### 6.4 向量嵌入服务
- 实现 LangChain 标准 Embeddings 接口
- 无缝对接 Milvus、RAG、检索链
- 使用阿里云 DashScope 文本嵌入模型 text-embedding-v4
- 输出 1024 维固定向量，与 Milvus 库维度一致
- 支持 `embed_query()` 和 `embed_documents()` 标准方法
- 懒加载 + 异常安全

### 6.5 智能 RAG 检索
完整流程：
1. 用户输入问题
2. 调用向量服务生成查询向量
3. 去 Milvus 做相似度检索
4. 返回最相关的 top_k 条文档
5. 交给 RAG Agent 生成最终答案

**检索配置：**
- 相似度算法：L2（欧氏距离），数值越小越相似
- 检索参数 `nprobe=10`，工业级常用参数，控制精度与速度
- 返回字段：id、content（文档片段）、score（相似度）、metadata（来源文件/章节）
